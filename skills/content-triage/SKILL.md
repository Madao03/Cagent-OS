# 内容分诊（Content Triage）

## ⚠️ 完成检查清单 — 每次分诊结束前自检

**以下三条全部为"是"，分诊才算完成。缺任何一条 = 没做分诊。**

- [ ] 我输出了五维分值（相关 X/2、新信息 X/2、判断 X/2、框架 X/2、来源 X/2）→ 总分 X/10 → A/B/C 类别？
- [ ] 每维我配了一句文中证据（引用具体事实/数据/来源）？
- [ ] 我用 `write.file`（mode="append"）把结果追加进了 **`knowledge/00_Inbox/分诊台账.md`**？

> 完成上面三条之后，分析格式、措辞、是否画表——你可以自由发挥。但数字和台账必须在。

---

把用户指定的一批内容按统一标准打分、分类、分流。目标是**省下注意力**：
让 A 类（值得进长期知识库的）被识别出来，B 类留要点不留全文，C 类直接丢弃。

裸提示词做这件事有三个固有短板，本技能针对性解决：

1. **打分随心情、不一致** → 给每个维度 0/1/2 的明确锚点，凭文中证据打分，不凭感觉。
2. **只给结论、不给依据** → 每个维度强制配一句「凭什么给这个分」，可复查、可被用户反驳改分。
3. **分完就散、无法回看** → 维护一份分诊台账，可周度回看「我把注意力花在了哪些标的/行业」。

---

## 硬约束（MUST — 不可跳过或变形）

以下三条是本技能的骨架。**跳过任何一条，分诊就失去了核心价值**：

| # | 约束 | 为什么 |
|:--|:-----|:-----|
| **1** | **必须输出五维分值和总分** | 没有数值，台账无法跨时间对比。"这篇文章很好"不是分诊 |
| **2** | **每维必须配一句文中证据** | 没有证据的分数不可复查、不可反驳、不可信 |
| **3** | **台账必须写入 `knowledge/00_Inbox/分诊台账.md`** | append-only 的前提是路径一致。两条路径 = 两个台账 = 废了 |

## ⚠️ 补充：文章长短与分诊无关

**文章长短不影响是否分诊。** 300 字和 30,000 字一样——都必须走完整的五维打分 + 台账追加流程。短文章可以更快完成打分（信息量小），但必须出五维分。一篇 2000 字的 Langfuse 介绍文，直接回答"这是关于 X 的文章"而不打分——这是错误的。必须是：

```
相关 X/2 | 新信息 X/2 | 判断 X/2 | 框架 X/2 | 来源 X/2 → 总分 X/10 → A/B/C
```

然后台账追加。**没有例外。**

## 软约束（推荐 — 具体呈现方式可自由发挥）

- 评分可以对话式呈现，不强制画 ASCII 表格。但数值和证据必须一眼可见
- 每篇结束可以给自然语言建议，不强制用模板字句
- 单篇文章可以不输出完整表头，但五维分 + 总分 + 类别（A/B/C）必须出现

## 开工前：确认两件事

1. **读重点清单**：用 `docs.read` 工具读取 `skills/content-triage/references/watchlist.md`（项目根目录下的相对路径）。这是「相关性」维度的唯一依据。**清单为空或缺失就先问用户**，不要凭空判断相关性。清单已由用户维护时，直接用。**绝对不要用 bash/cat/curl 读文件——本项目运行在 Windows 上，bash 不可用。**
2. **确认是否批量**：用户可能给 1 篇，也可能一次丢一批。批量时逐篇打分，最后汇一张总表。

## 五维打分锚点（每维 0 / 1 / 2）

逐篇按下表打分。**关键纪律：每个分都要在文中找得到证据**，把证据写出来（引用具体事实/数据/来源），否则分数不可信、也无法被反驳。

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| **相关性** | 与清单内标的/行业无关 | 间接相关：清单标的的**竞品、上下游、同赛道**，或宏观背景 | 直接命中清单内重点标的/赛道/主题 |
| **新信息** | 全是已知旧闻复述 | 有部分新数据/新视角，但增量有限 | 提供我此前不知道的关键新事实/一手数据 |
| **判断影响** | 不改变任何判断 | 边际补强或略微动摇某条现有判断 | 可能改变某个标的的持仓/估值/赛道判断 |
| **框架价值** | 无可复用方法 | 有可借鉴的分析角度，但偏情境 | 提供可长期复用的分析框架/思维模型 |
| **来源质量** | 匿名/情绪化/无数据/标题党；**或抽查数字核错的二手重构稿** | 一般媒体或二手分析，尚可信；**「现场实录/纪要」类公众号稿未经核验时至多停在这档** | 一手资料（财报/电话会/白皮书/权威深度研报）或可靠署名作者 |

打分要点：
- **「现场实录/会议纪要/电话会实录」类公众号稿默认按二手重构稿处理**：此类稿件常由记忆/网传素材重构，甚至含 AI 生成成分。打来源质量分前，**只在你怀疑数字有问题时才核查**——公开的百科全书事实（如"美联储资产负债表 $6.7T"）不需要验证，但声称独家数据（如"某基金仓位 30%""某 CEO 私下表态"）必须交叉验证。核查工具用 `web.fetch` 搜相关报道或 `financial.quote.verified` 核市场数据。
- **相关性看「对谁」而非「是谁」**。一篇讲阿里/美团的文章，若主体不在清单、但直接影响清单内的 PDD 竞争格局，相关性是 1（间接），而判断影响可能是 2——分流的真正理由要落在「它影响了我清单里的哪个标的」。
- **重复新闻 → 新信息直接 0**，通常落 C。**标题党/纯情绪/无数据 → 来源质量 0**，多半 C。
- 现状性数据（价格、份额、财务、政策）即便采纳也标「⚠️ 待核实」，不当既定事实。
- **⚠️"AI 技术"和"投资方法论"是独立学习锚点**：命中这两个主题的文章，相关性直接给 2 分。**不需要论证它"以后能否用来投资"**——用户正在学 AI 技术本身就是足够的相关性。一篇 Langfuse 观测框架的文章，不能说"它可以帮用户判断 AI 公司的产品是不是 demo，所以有用"——应该说"它命中 AI 技术主题 + 框架价值可复用 → A 类"。技术文章的价值 = 技术学习本身，不是投资决策的附庸。

## 判定与处理动作

总分 = 五维之和（0–10）。

| 总分 | 类别 | 处理动作 |
|---|---|---|
| **6–10** | **A 类 · 精读** | **本技能只做到：判定 + 标记「待精读」+ 写明关联标的与精读时该重点回答的问题**。精读本体（摘要/观点提炼/归档）由后续环节完成，本技能不展开。 |
| **3–5** | **B 类 · 速读** | 出一张「4 问速读卡」（见下），用 `write.file` 写入 `knowledge/00_Inbox/速读_<YYYY-MM-DD>_<标题简写>.md`。 |
| **0–2** | **C 类 · 丢弃** | 仅在台账记一行（标题 + 总分 + 一句理由），**不保留全文**。 |

### B 类「4 问速读卡」（固定四问）
1. 这篇文章说了什么？（2–3 句）
2. 它是否影响我的已有判断？（影响则点名哪条判断；不影响就明说）
3. 它关联哪些标的 / 行业？
4. 是否需要进一步研究？（是 → 给下一步建议；否 → 到此为止）

模板见 `references/templates.md`。

## 处理流程

**Step 1 · 读取**　逐篇读到结尾（长文/剪藏不能只读开头）。剥离广告、二维码、图片占位等噪音。URL 用 `web.fetch` 抓取，本地文件用 `docs.read` 工具。⚠️ 遇到 CDN 保护的机构研报站（Grayscale/Vercel 部署），`web.fetch` 会自动降级到浏览器模式；也可手动设 `browser_mode: true`。

**Step 2 · 逐维打分（MUST）**　按锚点表打 5 个分，**每维必须配一句文中证据**。这是不可跳过的步骤——没有证据的分数不可复查、不可信。

**Step 3 · 输出评分（MUST）**　将五维分值和总分呈现给用户。多篇时输出总表，单篇时至少包含：

```
相关: X/2 | 新信息: X/2 | 判断影响: X/2 | 框架价值: X/2 | 来源质量: X/2 → 总分 X/10 → A/B/C
```

每维下方紧跟一句证据。对话式呈现可以，但数值必须一眼可见。

**Step 4 · 临界裁决**　总分卡在 **5/6 边界**、或单维有争议的，单独标 ⚠️ 交用户裁决，**不自作主张升降级**。

**Step 5 · 给建议、等确认再执行**　默认**先把评分表 + 建议动作给用户**，等点头再落盘（写速读卡、记台账、标记待精读）。用户说「直接做」则一并产出。

**Step 6 · 维护台账（MUST）**　把本批结果追加进 **`knowledge/00_Inbox/分诊台账.md`**（没有就新建，表头见 `references/templates.md`）。**路径必须是 `knowledge/00_Inbox/分诊台账.md`**——不要用 `knowledge/分诊台账.md` 或其他路径。**必须使用 `write.file` 的 `mode: "append"` 参数追加**——只追加，不覆盖、不删旧行（知识库红线）。每次分诊必写台账，即使只有一篇 C 类。没有台账的分诊 = 没做分诊。

## 边界与红线
- **A 类不越界**：本技能不写精读笔记，只判定 + 标记。精读规范是独立环节。
- **不覆盖**：台账与速读卡同名文件已存在时，追加或加日期后缀，绝不覆盖。
- **台账 append-only**：每次用 `mode: "append"` 追加，**绝对不使用 overwrite 模式写台账**。
- **拿不准就问**：相关性无清单可依、或归属多个标的难取舍时，先问一句再动手。
- **诚实标临界**：5/6 边界如实标出交人裁决，不为了干脆而硬分。

## 一次好的分诊长什么样

### 必须出现（缺了就是没做分诊）
- [ ] 五维分值 + 总分 + A/B/C 类别——对话中一眼可见
- [ ] 每维配一句文中证据——可被用户反驳、改分
- [ ] 台账追加到 `knowledge/00_Inbox/分诊台账.md`——路径正确，mode=append

### 加分项
- [ ] 每个分都拿得出文中证据，用户能据此反驳或改分
- [ ] A/B/C 分得干脆，临界的诚实标出来交人裁决
- [ ] 分流理由落在「它影响我清单里的哪个标的/行业」，而不是「它讲了谁」
- [ ] B 类四问回答得能让人 5 秒决定要不要深挖
- [ ] 台账累积后，能看出自己一周把注意力放在了哪里
